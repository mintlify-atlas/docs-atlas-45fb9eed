Mandarin Talk is a real-time AI conversation app that streams raw microphone audio to Google's Gemini Live API and plays synthesized speech back through the browser. This page explains the full stack: how components are structured, how audio flows in both directions, and how the two Gemini function tools drive subtitles and pronunciation feedback.

## Tech stack

| Layer | Technology |
|-------|------------|
| Frontend framework | React 19 + Vite + TypeScript |
| 3D rendering | Three.js via `@react-three/fiber` and `@react-three/drei` |
| AI model | Gemini Live API — `gemini-2.5-flash-native-audio-preview-09-2025` |
| Styling | Tailwind CSS (loaded via CDN in `index.html`) |
| Icons | Lucide React |

## Audio pipeline

### Input: microphone to Gemini Live

<Steps>
  <Step title="Acquire the microphone stream">
    `LiveApiService.connect()` calls `navigator.mediaDevices.getUserMedia({ audio: true })` to get a `MediaStream`. If the device is not found or permission is denied, a fatal error is surfaced immediately.
  </Step>
  <Step title="Create a 16 kHz AudioContext">
    An `AudioContext` is created at exactly 16 000 Hz — the sample rate the Gemini Live API expects for PCM input.
  </Step>
  <Step title="Route audio through ScriptProcessorNode">
    The stream is connected: `MediaStreamAudioSourceNode` → `GainNode` (gain: 1.2) → `ScriptProcessorNode` (buffer size: 4 096 samples) → `AudioContext.destination`.
  </Step>
  <Step title="Downsample and buffer">
    Each `onaudioprocess` event calls `downsampleTo16k()` using linear interpolation. Downsampled chunks are accumulated until they reach the 4 096-sample threshold (256 ms of audio at 16 kHz), then merged with `concatenateFloat32Buffers()`.
  </Step>
  <Step title="Convert to PCM and send">
    `createPcmBlob()` converts `Float32Array` samples to little-endian `Int16` PCM and wraps the result in a `Blob` with MIME type `audio/pcm;rate=16000`. The blob is sent to the active Gemini session via `session.sendRealtimeInput({ media: pcmBlob })` through a serial send queue.
  </Step>
</Steps>

### Output: Gemini Live to speakers

<Steps>
  <Step title="Receive base64 audio chunks">
    Incoming `LiveServerMessage` objects are inspected for `serverContent.modelTurn.parts[0].inlineData.data`. When present, the base64 string is decoded to a `Uint8Array` by `base64ToBytes()`.
  </Step>
  <Step title="Decode to AudioBuffer">
    `decodeAudioData()` converts the raw `Int16` PCM bytes (24 000 Hz, mono) into a Web Audio API `AudioBuffer` by normalising each sample to the `[-1.0, 1.0]` float range.
  </Step>
  <Step title="Schedule playback">
    An `AudioBufferSourceNode` is connected through the `AnalyserNode` to the 24 000 Hz output `AudioContext`. Consecutive chunks are scheduled using a monotonically-increasing `nextStartTime` to produce gapless playback.
  </Step>
  <Step title="Feed volume to the avatar">
    A `requestAnimationFrame` loop reads `getByteFrequencyData()` from the `AnalyserNode`, averages all frequency bins, normalises to `[0, 1]`, and fires the `onAudioVolume` callback. `App.tsx` stores this value in state and passes it to `Avatar3D` every frame.
  </Step>
</Steps>

<Note>
When the server sends an `interrupted` signal, all scheduled audio sources are stopped immediately, the start-time cursor is reset, and the in-flight input transcription buffer is cleared.
</Note>

## Gemini function tools

The service registers two `FunctionDeclaration` objects with the Gemini session. The model calls them autonomously during conversation — no client-side parsing of the audio transcript is required.

### `update_subtitles`

Called by the model **while it is speaking** to provide a simultaneous written transcript.

```typescript
{
  name: 'update_subtitles',
  parameters: {
    hanzi:   string, // Chinese characters
    pinyin:  string, // Romanisation with tone marks
    english: string, // English translation
  }
}
```

When received, `LiveApiService` fires `onSubtitle(data: SubtitleData)`. `App.tsx` updates `currentSubtitle` state (driving `SubtitleDisplay`) and appends an `'ai'` entry to the conversation history.

### `provide_pronunciation_feedback`

Called by the model **after the user speaks Mandarin** to evaluate their pronunciation.

```typescript
{
  name: 'provide_pronunciation_feedback',
  parameters: {
    userText: string, // Chinese text the user spoke
    pinyin:   string, // Pinyin of what was heard
    score:    number, // 0–100 accuracy score
    feedback: string, // Constructive tip on tones or articulation
  }
}
```

When received, `LiveApiService` fires `onPronunciationFeedback(feedback: PronunciationFeedback)`. `App.tsx` updates `pronunciationFeedback` state, which renders `FeedbackCard`.

Both tools receive an acknowledgement response `{ result: 'ok' }` queued immediately after the callbacks fire.

## TypeScript interfaces

All shared data shapes are defined in `types.ts`.

```typescript
export interface SubtitleData {
  hanzi: string;
  pinyin: string;
  english: string;
}

export interface PronunciationFeedback {
  userText: string;
  pinyin: string;
  score: number;    // 0–100
  feedback: string;
}

export interface ConversationTurn {
  id: string;
  role: 'user' | 'ai';
  text: string;         // Hanzi for AI, recognised text for user
  pinyin?: string;
  translation?: string;
  timestamp: number;    // Unix ms
}

export interface Lesson {
  id: string;
  title: string;
  description: string;
  level: 'Beginner' | 'Intermediate' | 'Advanced';
  topic: string;
  scenario: string;     // System-prompt context block passed to LiveApiService
}

export enum ConnectionState {
  DISCONNECTED = 'DISCONNECTED',
  CONNECTING   = 'CONNECTING',
  CONNECTED    = 'CONNECTED',
  ERROR        = 'ERROR',
}

export interface AudioVolumeState {
  volume: number; // 0 to 1 — used internally for avatar animation
}
```

## Connection state machine

The `ConnectionState` enum drives all UI affordances — button labels, status pill colour, and microphone icon animation.

| State | Description |
|-------|-------------|
| `DISCONNECTED` | Default. No session active, microphone released. |
| `CONNECTING` | `connect()` called; waiting for WebSocket `onopen`. |
| `CONNECTED` | Session open; audio streaming active. |
| `ERROR` | Fatal error encountered; session stopped. |

## Error handling and recovery

`LiveApiService` applies a two-tier strategy:

1. **Transient network errors** — WebSocket and "Network error" / "Failed to fetch" strings are logged as warnings and silently ignored so a momentary blip does not interrupt the session.
2. **Internal errors** — The service attempts **one** automatic reconnect (`MAX_INTERNAL_ERROR_RETRIES = 1`). It stops the current session, resets all flags, emits a brief `"Reconnecting..."` notice, and calls `connect()` again after 500 ms using the same `scenarioInstruction` and `speakingRate`. If a second internal error occurs, it escalates to a fatal error.
3. **Fatal errors** — `handleFatalError()` sets a one-way `hasReportedFatalError` flag, fires `onError` and `onStateChange('ERROR')`, then calls `stop()` to fully tear down the session.
